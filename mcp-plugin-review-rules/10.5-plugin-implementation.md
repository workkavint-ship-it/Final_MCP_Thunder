## Notification and Sink Pattern

### Requirement

Plugin event subscriptions use a private inner `Notification` (or `*Notification`) class instantiated via `Core::SinkType<>`.

- The inner class must be delete-copy/move safe.
- Accept the parent by reference (not pointer) in the constructor; store as a reference.
- Declare `BEGIN_INTERFACE_MAP` / `END_INTERFACE_MAP` on the notification class.
- Instantiate as `Core::SinkType<Notification> _sink` (or named member) in the outer class — do NOT use `new`.
- Multiple notification types should each be separate inner classes, each with their own interface map.

### Example

```cpp
class NetworkControl : public PluginHost::IPlugin, public PluginHost::JSONRPC {
private:
    class NetworkNotification : public Exchange::INetworkControl::INotification {
    public:
        NetworkNotification() = delete;
        NetworkNotification(NetworkNotification&&) = delete;
        NetworkNotification(const NetworkNotification&) = delete;
        NetworkNotification& operator=(NetworkNotification&&) = delete;
        NetworkNotification& operator=(const NetworkNotification&) = delete;

        explicit NetworkNotification(NetworkControl& parent)
            : _parent(parent)
        {
        }
        ~NetworkNotification() override = default;

        void Update(const string& interface) override
        {
            Exchange::JNetworkControl::Event::Update(_parent, interface);
        }

        BEGIN_INTERFACE_MAP(NetworkNotification)
            INTERFACE_ENTRY(Exchange::INetworkControl::INotification)
        END_INTERFACE_MAP

    private:
        NetworkControl& _parent;
    };

private:
    Core::SinkType<NetworkNotification> _networkNotification;
};
```

## JSON Configuration Class

### Requirement

Plugin configuration parsed from `IShell::ConfigLine()` must be represented by an inner `Config` class inheriting from `Core::JSON::Container`.

- Must be delete-copy/move safe.
- Call `Add(_T("key"), &Member)` for every configuration field in the constructor.
- Declare fields as `Core::JSON::Boolean`, `Core::JSON::DecUInt16`, `Core::JSON::String`, `Core::JSON::ArrayType<>`, etc.
- Do **not** store the config object as a member — construct it on the stack in `Initialize()` and call `FromString(service->ConfigLine())` on it.

### Example

```cpp
class Config : public Core::JSON::Container {
private:
    Config(const Config&) = delete;
    Config& operator=(const Config&) = delete;

public:
    Config()
        : Core::JSON::Container()
        , Deferred(false)
        , Interval(30)
        , Retries(8)
        , Sources()
        , Periodicity(0)
    {
        Add(_T("deferred"),    &Deferred);
        Add(_T("interval"),    &Interval);
        Add(_T("retries"),     &Retries);
        Add(_T("sources"),     &Sources);
        Add(_T("periodicity"), &Periodicity);
    }
    ~Config() = default;

public:
    Core::JSON::Boolean                      Deferred;
    Core::JSON::DecUInt16                    Interval;
    Core::JSON::DecUInt8                     Retries;
    Core::JSON::ArrayType<Core::JSON::String> Sources;
    Core::JSON::DecUInt16                    Periodicity;
};
```

Usage in `Initialize()`:

```cpp
Config config;
config.FromString(service->ConfigLine());
uint16_t interval = config.Interval.Value();
```

## Thread Safety

### Requirement

- Protect shared mutable state with `Core::CriticalSection _adminLock`.
- Lock/unlock explicitly with `_adminLock.Lock()` / `_adminLock.Unlock()`, or use an RAII scope guard.
- Never call plugin methods that could re-enter while holding a lock.
- Do **not** perform long-running or blocking work on notification/callback threads — always dispatch to the worker pool.
- Background periodic jobs use `Core::WorkerPool::JobType<ClassName&> _job`; submit with `Core::IWorkerPool::Instance().Submit(...)`.

### Example

```cpp
Core::hresult TimeSync::Register(Exchange::ITimeSync::INotification* notification)
{
    _adminLock.Lock();
    _timeSyncObservers.push_back(notification);
    notification->AddRef();
    _adminLock.Unlock();
    return Core::ERROR_NONE;
}

void TimeSync::NotifyTimeChanged() const
{
    _adminLock.Lock();
    for (auto* observer : _timeSyncObservers) {
        observer->TimeChanged();
    }
    _adminLock.Unlock();
}
```

## Memory Management

### Requirement

Thunder uses **COM-style reference counting** — never use `std::shared_ptr` or `std::unique_ptr` for interface pointers.

- Call `AddRef()` when storing an interface pointer; call `Release()` when done.
- Immediately set the pointer to `nullptr` after calling `Release()`.
- Use `Core::ProxyType<T>` for reference-counted heap objects.
- Use `Core::ProxyPoolType<T>` for pre-allocated object pools (declare as `static` at file scope).
- Prefer stack allocation for temporaries; avoid raw `new` / `delete`.
- When releasing an OOP interface, check the return value against `Core::ERROR_CONNECTION_CLOSED` and `Core::ERROR_DESTRUCTION_SUCCEEDED` using `ASSERT`.

### Correct Example

```cpp
Exchange::INetworkControl* control =
    service->Root<Exchange::INetworkControl>(_connectionId, 2000, _T("NetworkControlImplementation"));

if (control != nullptr) {
    // ... use control ...
    VARIABLE_IS_NOT_USED uint32_t result = control->Release();
    control = nullptr;
    ASSERT((result == Core::ERROR_CONNECTION_CLOSED)
        || (result == Core::ERROR_DESTRUCTION_SUCCEEDED));
}

// ProxyType for JSON body objects
Core::ProxyType<Web::Response> result(PluginHost::IFactories::Instance().Response());

// ProxyPoolType declared at file scope
static Core::ProxyPoolType<Web::JSONBodyType<
    Core::JSON::ArrayType<Core::JSON::String>>> jsonStringListFactory(1);
```

### Incorrect Example

```cpp
// Do NOT use smart pointers for Thunder interfaces:
std::shared_ptr<Exchange::INetworkControl> control = ...;

// Do NOT use raw new/delete for interface objects:
Exchange::INetworkControl* control = new NetworkControlImpl();
delete control;
```
